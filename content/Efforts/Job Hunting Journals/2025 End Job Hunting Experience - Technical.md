这次面试前后我大概面了15多场，回头看，自己做得比较对的准备，主要有下面几件事。

**1. Coding 还是得准备，但不用神化它**  
前几轮基本都会考 LeetCode。我自己比较幸运，遇到的大多是 easy / medium。面试前大概集中刷了半个月，每天做几道，主要用 Python 写，因为确实会比 Java 更适合面试时快速表达思路。

**2. 项目经历一定要提前打磨**  
工作 1 年多后，项目经历的重要性非常高。  
我以前总觉得自己做的事情很零散，impact 也不够“大”，但后来发现关键不是项目本身有多厉害，而是你能不能把它讲清楚：背景是什么、问题是什么、你做了什么、为什么这样做、最后带来了什么结果。  
我平时有记录工作内容的习惯，所以在准备面试时，会把这些素材重新梳理，也会借助 ChatGPT 帮我把零散内容整理成更有逻辑的项目故事。

**3. 不只是准备答案，更要准备 follow-up**  
我觉得很有帮助的一点是，站在面试官的角度反过来问自己。  
比如：为什么这样设计？难点在哪？trade-off 是什么？如果重来一次会怎么改？  
你在准备时被“追问”得越充分，真正面试时就越稳。到后面会慢慢形成一种表达惯性，就算不是每句话都提前准备过，也能比较自然地把思路组织出来。

**4. 心态上，把面试当成 collaboration，会好很多**  
尤其是 project discussion、system design、一些 open-ended question。  
后来我会更倾向于把面试看成一次合作讨论，而不是单向被 challenge。对方很多时候并不是在“卡你”，而是在看你怎么思考、怎么沟通、怎么一起推进问题。这个 mindset 对我帮助还挺大的。

**5. Behavioral 和 AI usage 也值得准备**  
除了 coding 和项目，我这次还被问到不少 behavioral / open-ended 的问题。比较有意思的是，也有几次被问到工作中怎么使用 AI。感觉这已经慢慢变成一个挺常见的话题了，所以可以提前想一想：你平时具体怎么用、用来解决什么问题、边界是什么。


**还有一个小经验是：面挂了也不要太快放弃。**

在我最后拿到的 4 个 offer 里，其实有 2 家公司是一开始面试的岗位已经挂掉了。但当时我还是和 recruiter 保持了沟通，表达自己依然很希望加入这家公司，也想问问是否有其他更匹配的岗位机会。

后来 recruiter 确实帮我把简历 internal transfer 到了其他 team，重新走了一轮流程，最后也顺利拿到了 offer。

所以我这次一个很深的感受是：**面试挂掉不一定代表和这家公司完全结束。** 有时候只是某个岗位不太 match，如果你整体背景合适，其实可以稍微“脸皮厚”一点，多争取一下机会。当然沟通方式还是要真诚、礼貌。


整体来说，我觉得这次能比较顺，一方面当然有运气成分；但另一方面，提前把项目经历和表达方式准备好，确实会很大程度影响面试的手感。

## 求职复盘 Part 2｜技术面试（SWE / Backend）

接着上一篇，这一篇主要聊一下 **技术轮（technical interview）** 的准备和一些经验。

综合下来我自己的一个感受是：  
**PR + 1–2 年经验，在新加坡 tech 行业其实是一个比较好跳的阶段。**

这一篇主要讲 **SWE（backend）方向的技术面试**。

主要包括三个部分：

- 算法题（LeetCode）
    
- 项目经历（project discussion）
    
- 系统设计（system design）
    

---

# 1️⃣ 算法题（LeetCode）

除了 DevOps 岗位，**大多数 SWE 的岗位前几轮基本都逃不过 LeetCode**。

我自己比较幸运，遇到的题大多是 **easy / medium**。

面试前大概 **集中刷了半个月**，每天做几题，主要是找回手感。

### 一些 tips

**1️⃣ 强烈建议用 Python 写**

Python 的语法更简洁，而且有很多很好用的 built-in methods，可以省下很多 boilerplate code。

同样一个算法实现：

- Python 代码长度可能只有
    
- Java / C++ 的一半
    

这在面试中其实很重要，因为 **时间很紧张**。

代码写得越简洁，你越有时间解释思路。

---

**2️⃣ 不会 Python 也没关系，可以临时学**

我之前一直习惯写 Java。

但这次为了面试，我花了一两周集中练习用 Python 刷题，其实 **上手非常快**。

效果也很好：

能让我在更短时间内把思路表达清楚，  
省下来的时间还可以多聊项目经历。

---

# 2️⃣ 项目经历（Project Experience）

因为我已经有 **1.5 年工作经验**，所以面试中被问到很多 **project discussion**。

一开始准备面试的时候，我最大的一个困扰是：

> 总觉得自己做的事情都很零散，好像也谈不上什么 impact。

但后来慢慢发现，其实关键不是项目本身有多厉害，而是你能不能 **把它讲清楚**。

我后来基本按这个结构准备：

### 一个完整项目故事需要回答 4 个问题

1️⃣ **背景 / 问题是什么**

2️⃣ **你是什么角色，具体做了什么**

3️⃣ **为什么这样设计？背后的 rationale 是什么？有哪些 trade-off**

4️⃣ **最终带来了什么 impact**

---

### 一个很有用的方法：用 AI challenge 自己

对于 **1 和 2（背景 + 实现）**

通常可以根据你平时的工作笔记、文档梳理清楚。

但很多时候，真正难的是 **第 3 点：为什么这样设计**。

我当时会把项目背景和实现思路丢给 AI，让它反过来 **challenge 我**。

比如它会问：

- 为什么用 A 而不用 B？
    
- 如果系统某一块挂了，有没有 fallback / retry 机制？
    
- 有没有其他设计方案？
    
- 这个设计的 trade-off 是什么？
    

这些其实是我们自己准备的时候 **很难想到的问题**。

因为我们自己的思路是 **正向的**：

> “这个方案是合理的，所以我这么做。”

但 AI 很容易帮你提出 **why not 的问题**，  
更接近真实面试中的状态。

当你在准备的时候就已经被 challenge 过，  
真正面试时就会轻松很多。

---

### Impact 其实是 AI 最擅长的

至于 **impact（第 4 点）**，可以适当借助 AI 帮你整理。

AI 很擅长把一个看起来很普通的小项目：

- 结构化
    
- 总结价值
    
- 加一些具体例子
    

当然最后还是要自己改一改，保证真实。

---

# 3️⃣ 系统设计（System Design）

System design 在不同公司差异比较大。

我大概遇到过两类问题。

---

### 第一类：传统系统设计

比如：

- 设计一个 URL shortener
    
- 设计一个 rate limiter
    
- 设计一个 notification system
    

这类题主要考的是：

- 系统架构
    
- 数据流
    
- scalability
    
- trade-off
    

---

### 第二类：偏逻辑 / 偏实现的问题

有些公司会问更偏 **工程实现** 的问题，比如：

- 设计一个 message queue
    
- worker pool 怎么实现
    
- event loop 怎么设计
    

这种问题其实更接近 **系统内部机制**，  
有时候比传统 system design 还难一点。

---

### 对 junior 来说，不用太焦虑

对于 1–2 年经验的候选人，面试官其实更关注的是：

- 你的 **思维过程**
    
- 如何 **拆解问题**
    
- 是否能讨论 **trade-off**
    

而不是一定要给出一个完美答案。

很多时候面试更像是：

**你和面试官一起讨论一个系统。**

---

# 最后：准备几个 follow-up question

技术面试结束的时候，通常都会问：

> Do you have any questions for me?

建议 **提前准备一两个问题**。

不仅是为了礼貌，更是为了展示：

**你对技术是有 genuine curiosity 的。**

例如可以问：

- 团队现在最大的技术挑战是什么？
    
- 你们系统目前最复杂的部分在哪里？
    
- 新人加入后通常会先参与什么项目？
    

---

# 小总结

SWE 技术面试，大致可以拆成三块：

- **LeetCode / coding**
    
- **project discussion**
    
- **system design**
    

其中我自己感觉最重要的一点是：

> 项目经历一定要提前打磨好。

很多时候，技术面试不只是看你会不会写代码，  
更是看你 **是否真正理解自己做过的系统。**
